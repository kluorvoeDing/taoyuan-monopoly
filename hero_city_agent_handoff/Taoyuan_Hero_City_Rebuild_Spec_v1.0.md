# 桃園 Hero City 網頁遊戲重製規格書

**文件版本：** 1.0  
**基準原型：** `taofuweng_mvp_v0_7c`  
**用途：** 交付桌面型 Agent，作為從零重製的唯一功能規格與驗收依據  
**文件語言：** 繁體中文  
**目標平台：** 桌面瀏覽器優先，兼容一般筆電解析度  

---

## 0. 文件定位與優先順序

本規格不是要求 Agent 繼續修補既有的單檔 JavaScript 原型，而是要求其**重新建立可維護、可測試、可持續擴充的正式前端架構**。

執行優先順序如下：

1. 本文件中的「指定行為」與「驗收條件」。
2. 本文件附錄中的地圖、角色、卡片、事件資料。
3. `v0.7c` 原始碼僅作為介面與玩法概念參考。
4. 若原型程式與本文件不一致，**以本文件為準**。

禁止直接複製原型中的已知缺陷，例如：角色描述與實際效果不一致、`prompt()` 選目標、全域變數互相耦合、效果倍率散落在不同函式、舊版 localStorage key、卡片統計漏記等。

# 1. 產品摘要

## 1.1 遊戲概念

《桃園 Hero City》是一款以桃園地區為棋盤、融合地產經營、英雄角色能力、裝備卡、區域套裝及據點攻防的回合制網頁遊戲。

單局包含：

- 1 位真人玩家。
- 1～3 位 AI 玩家。
- 56 格環形棋盤。
- 10 個策略區域。
- 8 位角色。
- 地產進駐、擴建與支援費。
- 路線控制、據點防守、破壞與干擾。
- 支援裝備卡、突發事件、英雄／反派亂入。
- 以停業淘汰或回合結束時的聲望資產決定勝負。

## 1.2 核心體驗目標

遊戲不可只剩下「擲骰後被動接受結果」。玩家應能透過以下方式形成策略：

- 收集同一策略區域的據點，取得套裝加成。
- 阻止對手完成整區套裝。
- 使用移動卡控制落點與路線。
- 防守高價值據點。
- 破壞或干擾對手的關鍵據點。
- 管理現金、擴建時機、角色冷卻與手牌。

## 1.3 非目標

第一階段重製不包含：

- 即時多人連線。
- 房間密碼與 Lobby。
- 後端帳號系統。
- 競標機制。
- AP 行動點系統。
- 第二資源「英雄聲望」。
- 真實房價 API。
- 手機版完整體驗。

# 2. 法務與 IP 邊界

目前角色名稱與世界觀引用《我的英雄學院》，僅適合作為內部原型或非公開測試。

正式公開前必須支持「內容換皮」：

- 角色名稱、個性名稱與角色敘述必須由資料檔控制。
- 卡片與亂入角色名稱不得寫死在 UI 元件或遊戲引擎中。
- 遊戲邏輯只能依 `id` 與效果定義運作，不得依顯示名稱判斷。
- 需能替換成完全原創角色而不修改核心引擎。

- 遊戲狀態必須是可序列化的純資料。
- 遊戲動作以 Command／Action 表達。
- 引擎函式不得直接操作 DOM。
- 引擎函式不得呼叫 `alert()`、`prompt()` 或 `confirm()`。
- 隨機數必須透過可注入 RNG 介面產生，以便重播與測試。

- 同一個效果的文字描述與執行邏輯必須共用同一筆資料來源。

# 4. 遊戲流程與狀態機

## 4.1 頂層模式

```ts
type GameMode = 'setup' | 'playing' | 'finished';
type TurnPhase = 'preRoll' | 'resolvingMove' | 'action' | 'resolvingEffect';
```

## 4.2 標準回合

1. 進入玩家回合。
2. 在 `preRoll` 階段可：
   - 使用擲骰前卡片。
   - 使用允許在擲骰前發動的角色個性。
   - 直接擲骰。
3. 骰出點數或使用定點部署。
4. 移動並處理經過起點獎勵。
5. 依序處理：亂入標記 → 目前格效果。
6. 進入 `action` 階段。
7. 玩家可：
   - 進駐目前無主據點。
   - 擴建目前自己的據點。
   - 使用行動階段卡片。
   - 使用符合時機的角色個性。
   - 結束回合。
8. 回合結束時處理個人冷卻與個人狀態。
9. 一整輪結束時處理全域效果與據點狀態。
10. 切換至下一位未停業玩家。

## 4.3 停回合規則

「捕縛布」與「冰封戰線」的停止效果不可只跳過骰子。

正確行為：

1. 輪到被停止的玩家時，先再次觸發其目前所在格。
2. 若在他人據點，必須再次支付支援費。
3. 若在支援站、事件格、交通格或基金格，也再次觸發該格。
4. 處理完格子效果後，該玩家本回合不得移動或進行地產操作。
5. 切換下一位玩家。

為防止效果遞迴失控，同一個被停止回合只允許觸發一次所在格；交通連鎖移動需有最大深度保護。

## 4.4 冷卻語意

冷卻數字代表「還需等待幾個自己的完整回合」。

- 能力發動後設定冷卻 N。
- 每次該角色的新回合開始時減 1。
- 冷卻為 0 時才可再次使用。
- UI 顯示的數字必須與實際可用時機一致。

# 7. 經濟系統

## 7.1 土地價格

| 等級 | 基礎價格 |
|---|---:|
| S | 6,000 |
| A | 4,500 |
| B | 3,200 |
| C | 2,200 |

## 7.2 建築等級

| Level | 名稱 | 基礎支援費率 |
|---:|---|---:|
| 0 | 空據點 | 10% |
| 1 | 臨時據點 | 20% |
| 2 | 英雄事務所 | 35% |
| 3 | 大型事務所 | 55% |
| 4 | No.1 英雄事務所 | 80% |

## 7.3 擴建費

```text
基礎擴建費 = 土地基礎價格 × 50%
```

每次擴建費相同，再依角色、套裝、卡片與全域效果修正。

最低有效擴建費不得低於基礎擴建費的 10%。

## 7.4 支援費計算順序

```text
基礎支援費 = 土地基礎價格 × 建築等級費率
```

依序套用：

1. 全域支援費效果。
2. 土地擁有者角色被動。
3. 地區套裝加成。
4. 據點干擾或停擺。
5. 擁有者的亂入／狀態收益加成。
6. 支付者的亂入／狀態懲罰。
7. 一次性英雄廣告看板，×1.5。
8. 一次性無重力漂浮，-50%。
9. 硬化防守、敵人預警等支付折扣。
10. 麗日御茶子 20% 機率再打 8 折。
11. 四捨五入為整數。

支援費不得低於 0。

## 7.5 起點

- 經過英雄總部：獲得 2,000。
- 精確停在英雄總部：額外獲得 1,000。
- 因此從尾端跨越並精確停在起點，可總計獲得 3,000。

## 7.6 交通格

- 一般交通費：300。
- 飯田天哉支付交通費 +20%。
- 飯田抵達交通格後額外前進 2 格，且觸發新格。
- 額外移動不重複發放經過起點獎勵。
- 交通連鎖解析最大深度建議為 8，超過時中止並記錄錯誤。

## 7.7 支援基金

指定重製規則：

- 參加費：500。
- 參加費全額加入基金池。
- 20% 機率中獎。
- 中獎獲得當前基金池 70%。
- 未中獎時基金池保留。
- 資金不足 500 時不參加。

此規則修正原型中「檢查 500、實際扣 800、基金只加 500」的不一致。

## 7.8 聲望資產

聲望資產只是結算分數，不是第二種可消耗資源。

```text
聲望資產 = 現金
          + 所有土地基礎價格
          + Σ（土地等級 × 土地基礎價格 × 35%）
```

## 7.9 緊急撤收與停業

當現金低於 0：

1. 依固定、可預測的順序出售名下據點，建議由低策略價值到高策略價值；不可依陣列偶然順序。
2. 單一據點回收金：

```text
土地基礎價格 × 50%
+ 建築等級 × 土地基礎價格 × 25%
```

3. 撤收時清除所有土地狀態。
4. 若出售後仍為負數，玩家停業。
5. 停業玩家現金歸 0、全部土地釋出、跳過後續回合。

# 8. 地產操作規則

- 每回合原則上只能進行 1 次地產操作。
- 地產操作包括：進駐或擴建。
- 抵達無主土地後才可進駐該土地。
- 抵達自己的土地後才可擴建該土地。
- 土地最高 Level 4。
- 臨時執照可重置一次地產操作，但同一玩家同一回合只能實際使用 1 張臨時執照。
- 進駐或擴建成功後必須立即更新 UI、資產與套裝進度。

- 效果成功提交後才移除卡片。
- 不可用 `prompt()` 選目標，必須使用遊戲內目標選擇器。
- 目標土地需要顯示所有者、等級、區域、套裝威脅及狀態。
- 卡片使用必須寫入事件紀錄與統計。

# 13. 據點攻防狀態

| 狀態 | 欄位 | 效果 | 衰減 |
|---|---|---|---|
| 防守 | `guardTurns` | 抵銷一次破壞或干擾，觸發後清零 | 每完整輪 -1 |
| 干擾 | `disruptedTurns` | 收取支援費 ×0.6 | 每完整輪 -1 |

「一輪」指所有仍在遊戲中的玩家各完成一次回合，而不是單一玩家回合。

# 14. AI 規格

## 14.1 基本行為

AI 必須使用與真人相同的 Command 與驗證規則，不得直接修改狀態繞過卡片限制。

## 14.2 地產評分

建議評分：

```text
土地評分 = 等級基礎分
         + 自己在該區已持有數 × 18
         + 即將完成整區時額外 +70
         + 若是阻止對手成套額外 +80
```

等級基礎分：S 50、A 36、B 24、C 16。

## 14.3 決策優先順序

1. 避免停業並保留最低現金緩衝。
2. 完成自己的區域套裝。
3. 阻止對手取得最後一塊套裝地。
4. 擴建已成套或高收益土地。
5. 防守最高價值或最關鍵據點。
6. 對對手即將成套或高等級土地使用破壞／干擾。
7. 依前方 1～6 格選擇高速或低速路線卡。
8. 資金不足時使用現金卡或維修卡。

## 14.4 AI 可觀測資訊

目前 AI 可使用完全資訊，不需要隱藏手牌。但架構需允許未來改為有限資訊。

# 15. UI／UX 規格

## 15.1 全局布局

- 頂部：遊戲標題、簡短版本說明、重新開局。
- 主體：固定視窗範圍內的 15×15 棋盤。
- 中央：13×13 可用區域，不能留下大片空白，也不可把資訊過度壓縮在左上角。
- 桌面 1440×900 以上應完整顯示，整頁不產生水平捲軸。

## 15.2 中央區

中央採兩欄兩列：

- 左上：目前回合、骰子、主要操作。
- 右上：手牌、角色能力、全域效果、亂入、平衡測試。
- 左下：2×2 玩家卡。
- 右下：常駐事件紀錄。

操作按鈕依階段突出：

- 擲骰前：擲骰為主要按鈕。
- 行動階段：結束回合為主要按鈕。
- 無法使用的按鈕應 disabled，但不得消失造成版面跳動。

## 15.3 地圖格視覺層級

每格常駐只顯示：

1. 格子 ID。
2. 短名稱。
3. 土地級別與簡寫價格，或特殊格類型。
4. 上緣策略區域色帶。
5. 右側所有權色條。
6. 右上建築圖示。
7. 右下玩家棋子。
8. 防守／干擾／停擺圖示。

要求：

- 策略區域不得再用左下角短色條，避免與文字重疊。
- 土地名稱單行省略，但完整名稱在 tooltip 與點擊彈窗顯示。
- 所有權色與策略區域色必須是兩個不同視覺通道。
- Hover 要有明確可點擊回饋。
- Selected 格保留高亮外框。

## 15.4 格子資訊彈窗

點擊格子顯示：

- 完整名稱、副標與 ID。
- 格子類型。
- 行政區與策略區域。
- 土地級別、建築等級。
- 基礎價格、當前進駐價、擴建價、支援費。
- 所有者與所在玩家。
- 套裝進度與當前加成。
- 防守、干擾、停擺與廣告狀態。
- 亂入資訊。

彈窗可由 X、點擊遮罩或 Escape 關閉。

## 15.5 玩家卡

2×2 固定排列，不需內部垂直捲軸。每張卡顯示：

- 玩家色點、角色名、真人／AI。
- 即時排名；第一名皇冠。
- 現金、土地數、完成套裝數、聲望資產、手牌數、個性冷卻。
- 目前位置、最接近完成的區域、簡短狀態。
- 當前玩家用外框高亮。
- 停業玩家半透明。
- 完整角色能力以 tooltip 或詳情彈窗呈現。

## 15.6 事件紀錄

- 永久常駐。
- 區塊內獨立垂直捲動，不增加整頁高度。
- 新事件置頂。
- 最多保留 120 筆。
- 玩家名稱使用對應玩家色與粗體。
- 正數資金綠色、負數資金紅色。
- 每筆以短句呈現，不顯示「原本」「修改為」等開發紀錄。

## 15.7 選擇器

不得使用瀏覽器原生 `prompt()`：

- 角色指定：玩家選擇彈窗。
- 土地指定：可篩選的土地清單或直接在棋盤選取。
- 骰子指定：1～6 的視覺按鈕。
- 交通格指定：交通節點卡片。

取消選擇時，不消耗卡片、資金或冷卻。

# 16. 資料模型

```ts
interface GameState {

  raidSpawns: RaidSpawn[];

  eventLog: GameLogEntry[];
  selectedTileId?: number;
  stats: GameStats;
  rngState?: string;
}

interface PlayerState {
  id: string;
  characterId: string;
  control: 'human' | 'ai';

```

# 17. Command 與事件模型

建議 Command：

```ts
type GameCommand =
  | { type: 'START_GAME'; options: GameOptions; characterId: string }
  | { type: 'ROLL_DICE'; playerId: string }
  | { type: 'USE_ABILITY'; playerId: string; payload?: unknown }
  | { type: 'USE_CARD'; playerId: string; cardInstanceId: string; payload?: unknown }
  | { type: 'BUY_CURRENT_TILE'; playerId: string }
  | { type: 'UPGRADE_CURRENT_TILE'; playerId: string }
  | { type: 'END_TURN'; playerId: string }
  | { type: 'SELECT_TILE'; tileId: number };
```

引擎輸出：

```ts
interface CommandResult {
  state: GameState;
  events: DomainEvent[];
  error?: GameRuleError;
}
```

所有資金變動、狀態變動、卡片消耗與地產轉移都應產生 Domain Event，供事件紀錄、動畫、統計與未來多人同步使用。

# 19. 測試規格

## 19.1 單元測試必測項目

### 經濟

- 各級土地價格與支援費。
- 各建築等級費率。
- 套裝門檻與完成整區。
- 多重加成的計算順序。
- 最低擴建費限制。
- 聲望資產公式。
- 緊急撤收與停業。

### 角色

每位角色至少測試：

- 效果可使用條件。
- 成本不足。
- 取消目標。
- 冷卻開始與結束。
- 被動與其他效果疊加。

特別測試：

- 綠谷所有預算模式起始資金與其他玩家完全相同。
- 爆豪 Level 3 發動主動時最多升至 Level 4，不可溢位。
- 轟使對手停回合時，對手在他人土地上再次繳費。
- 麗日主動與被動同時發生時的折扣順序。

### 卡片

23 張卡片每張至少：

- 正常使用。
- 非法時機。
- 非法目標。
- 取消選擇。
- 效果成功後卡片才消耗。
- AI 使用與真人使用結果一致。

### 路線與遞迴

- 經過起點與停在起點。
- 飯田交通格額外前進。
- 定點部署再次觸發。
- 捕縛／冰封再次觸發。
- 交通連鎖不造成無限遞迴。

## 19.2 整合測試

- 20,000／30 回合標準局完整跑完。
- 400,000／180 回合壓力局不溢位、不變慢、不產生 NaN。
- 1、2、3 位 AI 均可完成遊戲。
- 亂入開啟與關閉。
- 所有玩家除一人外停業。
- 回合數結束時依聲望資產排序。

## 19.3 UI E2E

- 設定完成後中央區不可空白。
- 所有按鈕依 phase 正確 enabled／disabled。
- 56 格無重疊、無裁切至無法辨識。
- 點擊每種格子可開啟正確彈窗。
- 玩家卡不出現垂直捲軸。
- 事件紀錄只在自身區域捲動。
- 在 1366×768、1440×900、1920×1080 測試。

# 20. 已知原型缺陷：重製時不得保留

1. 曾出現綠谷起始資金 `startingCash × 1.1`；正式規則禁止。
2. 角色與卡片描述曾與執行值不同，例如 -50% 寫成免付、×1.5 實作成 ×2。
3. 原型使用 `prompt()` 選目標。
4. `getZoneTiles()` 曾存在重複 `return`。
5. 支援基金曾檢查 500、扣 800、池只加 500。
6. 據點干擾狀態曾設定 2 輪但 log 顯示 3 輪。
7. 卡片使用統計不是每張都一致記錄。
8. AI 可能直接改狀態，繞過卡片驗證。
9. 冷卻在回合結束扣減，容易造成顯示與實際等待回合不一致。
10. keydown listener 在重繪時可能重複註冊。
11. localStorage key 使用舊版名稱 `taofuwengOptionsV06e`。
12. 靜態資料、規則與 UI 混在同一檔案，不可沿用。
13. 遊戲內文案不得顯示「原本」「移除」「不再」等開發修改紀錄。
